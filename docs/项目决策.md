## 一、已确立的决策

### D-001：前端框架选型

- **决策**：SvelteKit + Svelte 5
- **理由**：前后端一体，开发效率高，学习成本低
- **状态**：✅ 已实施（v0.1.0）

### D-002：节点编辑器选型

- **决策**：LiteGraph.js（npm 原版，v0.7.18）
- **理由**：ComfyUI 同款引擎，开箱即用
- **已知风险**：最后更新 2023 年，长期维护风险大
- **状态**：✅ 已实施（v0.1.0），待评估替代方案（见 D-007）

### D-003：桌面壳子选型

- **决策**：Tauri 2.x
- **理由**：体积小（13.6MB vs Electron 150MB+）、Rust 后端性能好、不需要 Node.js 运行时
- **状态**：✅ 已实施（v0.5.0）

### D-004：后端从 Node.js 迁移到 Rust

- **决策**：所有 API 逻辑移至 Rust（chat、agent、open_file）
- **理由**：静态打包后无 Node.js 运行时，Rust 编译为原生代码启动快、内存小
- **状态**：✅ 已实施（v0.5.0）

### D-005：静态导出替代 SSR

- **决策**：使用 adapter-static，前端纯静态文件
- **理由**：Tauri 加载本地文件，不需要服务端渲染，所有 API 在 Rust 后端
- **状态**：✅ 已实施（v0.5.0）

### D-006：Agent 事件推送方案

- **决策**：Tauri `app.emit()` + 前端 `listen()`
- **替代方案**：SSE 流（需要 Node.js 运行时）
- **理由**：进程内通信延迟低，不依赖 Node.js
- **状态**：✅ 已实施（v0.5.0）

### D-007：文件产物节点按钮绘制

- **决策**：Canvas 自定义绘制（不用 LiteGraph 内置按钮）
- **理由**：LiteGraph 内置按钮 `textBaseline` 居中有 bug，无法修复
- **状态**：✅ 已实施（v0.5.0）

### D-008：AI API 格式

- **决策**：OpenAI 兼容格式（DeepSeek V4）
- **理由**：兼容性强，切换模型成本低
- **状态**：✅ 已实施（v0.1.0）

## 二、待决策问题

### D-007：节点图库是否需要替换

- **问题**：LiteGraph.js 停更（最后更新 2023），npm 原版功能有限
- **候选方案**：
  - xyflow（原 React Flow，活跃维护，社区大）
  - Rete.js（框架无关，插件生态）
  - 继续使用 LiteGraph（稳定，够用）
- **影响范围**：整个前端画布层，是重大重写
- **当前优先级**：P3
- **各方结论**：
  - 🏗️ 架构：暂不替换。当前功能完整，LiteGraph 是纯 Canvas 绘制与 Svelte 无耦合，停更不影响当前功能。迁移成本 2-4 周，收益不确定。建议 fork 一份到本地应急。
  - 🎯 产品：P3 优先级，当前够用。
- **决策状态**：✅ 已决策 — **暂不替换，保持关注，fork 到本地**

---

### D-008：是否引入 TypeScript

- **问题**：全部 JS，无类型安全，维护成本随规模增长
- **各方结论**：
  - 🏗️ 架构：渐进式引入。新文件用 TS，旧文件不动。优先给 IPC 接口加类型（`ChatRequest`、`AgentEvent`）。v0.6.0 新增文件用 TS，v0.7.0 IPC 接口类型化，v0.8.0+ 旧文件按需迁移。
  - 🎯 产品：单人开发 ROI 低，P3。
- **决策状态**：✅ 已决策 — **渐进式引入，新文件从 v0.6.0 起用 TS**

---

### D-009：API Key 安全存储方案

- **问题**：当前 API Key 存在前端内存中，明文，不安全
- **各方结论**：
  - 🔍 QA：5 个 P0 安全漏洞，API Key 明文写磁盘、泄露到导出 JSON、HTTP 明文传输。
  - 🏗️ 架构：分步实施。v0.6.0 用 `aes-gcm` 文件加密（密钥 derived from machine-id），v0.7.0 迁移到系统钥匙串。`tauri-plugin-stronghold` 尚不稳定，暂不使用。
  - 🎯 产品：可分步，先明文持久化再加密。
- **决策状态**：✅ 已决策 — **v0.6.0 文件加密 → v0.7.0 系统钥匙串**

**一、安全审计**

**1.1 API Key 安全（严重程度：🔴 高）**

传递路径分析：
```
前端 ModelConfigNode._apiKey
  → stores/app.js _config.apiKey（内存明文）
  → invoke('chat', { request: { apiKey } })（IPC 明文）
  → lib.rs ChatRequest.api_key（Rust 内存明文）
  → Authorization: Bearer {api_key}（HTTP Header 明文）
  → save_config() → config.json（磁盘明文）
  → exportGraph() → _apiKey 序列化到导出 JSON
```

问题清单：
| # | 问题 | 位置 | 严重程度 |
|---|------|------|----------|
| S-01 | API Key 明文写入磁盘 config.json | `lib.rs:750-758` | 🔴 高 |
| S-02 | API Key 被序列化到导出的对话 JSON | `io.js:15` | 🔴 高 |
| S-03 | 无 HTTPS 强制，HTTP base_url 时 Key 在网络明文传输 | `lib.rs:357` / `+server.js:29` | 🔴 高 |
| S-04 | CSP 完全禁用（`"csp": null`） | `tauri.conf.json:26` | 🟡 中 |
| S-05 | config.json 文件权限未限制（任何本地程序可读） | `lib.rs:744-747` | 🟡 中 |

修复建议：
- [ ] P0：API Key 不写入导出 JSON，导出前清除敏感字段
- [ ] P0：base_url 为 http:// 时弹出安全警告
- [ ] P1：使用 Tauri stronghold 插件或系统 Keychain 存储 Key
- [ ] P1：收紧 CSP 策略，至少限制 script-src

**1.2 命令注入（严重程度：🔴 高）**

漏洞 1：`execute_command` 无任何防护
```rust
// lib.rs:276-279 — Rust 端
Command::new("cmd")
    .args(["/C", &cmd])  // cmd 来自 LLM tool_call，可被 prompt injection 操控
    .current_dir(work_dir)
    .output()
```
```javascript
// index.js:159 — Node.js 端
execAsync(args.command, { cwd: WORK_DIR, timeout: 30000 })
```
LLM 可被诱导执行任意系统命令（`del /f /s /q C:\*`、`format`、`regedit` 等），等同于给 LLM 一个 root shell。

漏洞 2：`search_in_files` 引号逃逸
```rust
// lib.rs:297
format!("findstr /S /N /C:\"{query}\" {search_path}\\*.js ...")
```
若 `query` 包含 `"` 字符可闭合引号，注入任意命令。

漏洞 3：Node.js 端 `search_in_files` 同样存在注入
```javascript
// index.js:174-176
`grep -rn "${args.query}" "${searchPath}" ...`
```
`query` 或 `searchPath` 包含 `"` 或 `$()` 即可注入。

| # | 问题 | 位置 | 严重程度 |
|---|------|------|----------|
| S-06 | execute_command 无命令白名单/黑名单 | `lib.rs:271-288` / `index.js:158-168` | 🔴 高 |
| S-07 | search_in_files 命令注入（Rust 端） | `lib.rs:297` | 🔴 高 |
| S-08 | search_in_files 命令注入（Node.js 端） | `index.js:174` | 🔴 高 |

修复建议：
- [ ] P0：execute_command 增加危险命令黑名单（rm -rf、format、regedit、shutdown 等）
- [ ] P0：execute_command 增加用户确认机制（首次调用弹窗确认）
- [ ] P0：search_in_files 使用参数化调用而非字符串拼接
- [ ] P1：考虑沙箱隔离（Windows AppContainer 或 Docker）

| 维度 | 评分 | 说明 |

**1.2 前后端职责划分**

划分合理，具体分工：

| 职责 | 归属 | 评价 |
|------|------|------|
| 画布渲染、节点交互 | 前端（Canvas API） | ✅ 合理，LiteGraph 是纯 Canvas 库 |
| AI API 调用 | Rust（reqwest） | ✅ 合理，避免前端暴露 Key |
| Agent 工具执行 | Rust（std::fs / Command） | ✅ 合理，安全边界在 Rust 端 |
| 事件推送 | Rust → 前端（Tauri emit） | ✅ 合理，比 SSE 延迟低 |
| 配置持久化 | Rust（serde_json + 文件） | ✅ 合理 |
| 对话历史回溯 | 前端（graph.links 遍历） | 🟡 可接受，但逻辑较脆弱 |

**问题**：Node.js 端（`src/lib/tools/index.js`、`src/routes/api/`）与 Rust 端存在**完整的功能重复**。Node.js 端用于开发模式（`npm run dev`），Rust 端用于生产（Tauri 打包）。两套代码的工具定义、执行逻辑、安全检查各自独立，维护成本翻倍。

**建议**：v0.7.0 考虑删除 Node.js 端 API 路由，开发模式也走 Tauri（`cargo tauri dev` 已支持热重载），消除双端维护。

**1.3 节点系统扩展性**

**当前扩展成本：添加 1 种新节点需改 6-7 个文件**

| 步骤 | 文件 | 改动 |
|------|------|------|
| 1. 创建节点定义 | `XxxNode.js`（新建） | 注册节点、定义端口、绘制逻辑 |
| 2. 导入注册函数 | `init.js` | import + registerXxxNode(LiteGraph) |
| 3. 加入白名单 | `init.js` allowedTypes | 添加类型名 |
| 4. 缩略图颜色 | `init.js` minimap.nodeColors | 添加颜色映射 |
| 5. 事件处理 | `agent.js` | 添加 case 分支、创建节点、连线 |
| 6. 端口定义文档 | `开发文档.md` | 更新表格 |

**痛点**：
- `allowedTypes` 白名单硬编码，忘记添加则节点不可用
- `minimap.nodeColors` 与节点定义分离，容易遗漏
- `agent.js` 的 switch-case 是平铺结构，10+ case 后难以维护

**建议**：引入节点注册表模式——每个节点文件导出 `{ type, color, register }`，`init.js` 自动收集注册，消除手动同步。

**1.4 Agent 系统扩展性**

| 4. Node.js 执行逻辑 | `index.js` executeTool() | 添加 case 分支 |
| 5. 工具描述优化 | `lib.rs` + `index.js` | 双端同步修改 |

**痛点**：

**理由**：
- `tauri-plugin-stronghold` 尚在开发中，API 不稳定
- 系统钥匙串需要平台特定代码（Windows Credential Manager / macOS Keychain / Linux Secret Service）
- 分步实施降低风险：先解决"明文写磁盘"的最坏情况，再升级到系统级安全

**v0.6.0 方案（文件加密）**：
- 用 `aes-gcm` crate 加密 API Key，密钥 derived from 设备唯一标识（machine-id）
- `save_config` 写加密后的密文，`load_config` 解密
- 导出 JSON 时清除 `apiKey` 字段

**v0.7.0 方案（系统钥匙串）**：
- Windows：`windows-sys` crate 调用 `CredWrite` / `CredRead`
- macOS：`security-framework` crate 调用 Keychain API
- Linux：`secret-service` crate 调用 GNOME Keyring

**代价**：v0.6.0 约 2 天；v0.7.0 约 3 天

**D-010：安装包与分发方案**

**建议：用 NSIS**

**理由**：
- Tauri 官方支持 NSIS 作为 Windows 打包目标
- NSIS 跨平台（可在 Linux/macOS 交叉编译 Windows 安装包）
- 体积小（NSIS 安装包仅比 exe 大几百 KB）
- 支持自定义安装流程（选择安装目录、创建桌面快捷方式、卸载程序）

**替代方案对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| NSIS | 跨平台、体积小、Tauri 原生支持 | 需要学习 NSIS 脚本 |
| MSIX | Windows 原生、支持自动更新 | 需要代码签名证书、不支持 Win7 |
| 裸 exe | 零成本 | 不专业、无卸载程序 |

**代价**：1 天配置 NSIS 脚本（Tauri 文档有示例）

**建议：用 Tauri 内置 updater + GitHub Releases**

**理由**：

**代价**：2-3 天（配置 updater 插件 + GitHub Actions 自动发布）
