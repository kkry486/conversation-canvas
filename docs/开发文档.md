# 对话画布（Conversation Canvas）开发文档

> 版本：v0.5.0
> 日期：2026年6月19日
> 状态：节点图对话 + Agent 工具调用 + Tauri 桌面应用

---

## 1. 项目概述

对话画布是一个基于节点图的 AI 对话系统，借鉴 ComfyUI 的交互方式，将传统线性对话改造为可视化节点编辑器。v0.5.0 新增 Agent 模式——AI 可以自动调用本地工具（读写文件、执行命令），节点图实时展示 Agent 的每一步思考过程。

### 技术栈

| 层 | 技术 | 版本 |
|---|------|------|
| 前端框架 | SvelteKit + Svelte 5 | 2.x |
| 节点编辑器 | LiteGraph.js | 0.7.18 |
| 构建工具 | Vite | 6.x |
| 桌面壳子 | Tauri | 2.x |
| Rust 后端 | reqwest + tokio | - |
| AI API | OpenAI 兼容格式 | DeepSeek V4 |

## 3. 架构设计

### 3.1 数据流

```
对话模式：
[模型配置] → [对话输入] → invoke('chat') → Rust → DeepSeek API → 返回文本

Agent 模式：
[模型配置] → [对话输入] → invoke('agent') → Rust Agent 循环
                                                    ↓
                                          LLM → tool_use → 执行工具
                                                    ↓
                                          app.emit('agent-event') → 前端监听
                                                    ↓
                                    创建 🧠/🔧/📋/📄 节点
```

### 3.2 Agent 循环

```
用户输入
    ↓
构建 messages 数组（system + history + user）
    ↓
┌──→ 调用 LLM（带 tools 定义）
│        ↓
│   有 tool_calls？──否──→ 完成，返回文本
│        ↓ 是
│   执行工具（read_file / write_file / ...）
│        ↓
│   推送 tool_call + tool_result 事件
│   工具结果加入 messages
│        ↓
└── 循环（最多 10 轮）
```

### 3.3 节点端口定义

| 节点 | 输入 | 输出 |
|------|------|------|
| 模型配置 | 无 | 配置(object) |
| 对话输入 | 配置(object), 上下文(string) | 消息(string), 配置(object) |
| AI 回复 | 回复(string) | 内容(string) |
| Agent/思考 | 输入(string) | 思考(string) |
| Agent/工具调用 | 触发(string) | 结果(string) |
| Agent/工具结果 | 触发(string) | 结果(string) |
| Agent/文件产物 | 触发(string) | 文件(string) |

### 3.4 工具列表

| 工具 | 功能 | 参数 |
|------|------|------|
| read_file | 读取文件 | path |
| write_file | 写入文件（返回完整路径） | path, content |
| list_directory | 列出目录 | path |
| execute_command | 执行 shell 命令 | command |
| search_in_files | 搜索文件内容 | query, path |

## 4. Tauri 启动器

### 4.1 为什么选 Tauri

| 方案 | 体积 | 性能 | 学习成本 |
|------|------|------|----------|
| Tauri | ~5-15MB | 高（Rust 后端） | 中 |
| Electron | ~150MB | 中（Node.js） | 低 |

### 4.2 Rust 后端命令

```rust
// 调用 LLM
#[tauri::command]
async fn chat(request: ChatRequest) -> Result<String, String>

// Agent 循环（通过事件推送进度）
#[tauri::command]
async fn agent(request: ChatRequest, app: AppHandle) -> Result<(), String>

// 打开文件/文件夹
#[tauri::command]
fn open_file(path: String, mode: String) -> Result<(), String>

// 设置工作目录
#[tauri::command]
fn set_work_dir(path: String)
```

### 4.3 前端调用方式

```javascript
import { invoke } from '@tauri-apps/api/core';
import { listen } from '@tauri-apps/api/event';

// 对话模式
const text = await invoke('chat', { request: { model, apiKey, baseUrl, message, history } });

// Agent 模式
const unlisten = await listen('agent-event', (event) => {
  const ev = event.payload;
  // ev.type: 'thinking' | 'text_delta' | 'tool_call' | 'tool_result' | 'done' | 'error'
});
await invoke('agent', { request: { model, apiKey, baseUrl, message, history } });
unlisten();

// 打开文件
await invoke('open_file', { path: 'C:\\...', mode: 'file' });
```

### 4.4 打包

```bash
npm run build                          # 构建前端静态文件
cargo tauri build                      # 编译 Rust + 打包
# 产物：src-tauri/target/release/conversation-canvas.exe
```

## 6. 关键技术决策

### 6.1 Agent 事件推送方案

**选择**：Tauri `app.emit()` + 前端 `listen()`

**替代方案**：SSE 流（Node.js 后端）

**原因**：静态打包后没有 Node.js 运行时，SSE 不可用。Tauri 的事件系统是进程内通信，延迟更低。

### 6.2 文件产物节点按钮

**选择**：Canvas 自定义绘制

**替代方案**：LiteGraph `addWidget('button')`

**原因**：LiteGraph 内置按钮文字垂直居中有 bug（`textBaseline` + `+1` 偏移），无法修复。自定义绘制用 `textBaseline: 'middle'` 精确居中。

### 6.3 Rust 后端 vs Node.js 后端

**选择**：Rust

**原因**：
- Tauri 打包不需要 Node.js 运行时
- Rust 编译为原生代码，启动快、内存小
- reqwest HTTP 库性能好

### 6.4 静态导出 vs SSR

**选择**：静态导出（adapter-static）

**原因**：Tauri 加载本地文件，不需要服务端渲染。所有 API 逻辑在 Rust 后端处理。
